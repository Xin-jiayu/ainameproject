# AI 智能命名平台项目规划文档

## 2026-07-02 架构更新：Service 层与 RabbitMQ 异步任务

当前后端已完成一次架构升级：从“Router + Repository + Core”的功能型结构，演进为“Router + Service + Repository + Worker + Infrastructure”的分层结构。

已完成内容：

- 新增 `ainame/services/` 业务服务层。
- `NameService` 承接起名、反馈、候选名保存、使用记录、异步任务提交和任务执行。
- `UserService` 承接额度扣减、退款、冻结判断、权益查询和管理员用户操作。
- `KnowledgeService` 承接知识库上传、删除、失败重试和 RabbitMQ 投递。
- `OrderService` 承接订单创建、查询和模拟支付。
- `ValidationService` 承接候选名、域名、商标、社媒校验。
- `AIService` 承接 `core.workflow` 的业务封装。
- `EmailService` 承接邮件投递和邮件发送。

RabbitMQ 已确定为统一异步任务队列，不再引入第二套队列体系。

当前队列划分：

```text
rag_document_queue      -> 知识库文件解析
name_generation_queue   -> AI 起名任务
email_queue             -> 验证码邮件/测试邮件
```

AI 起名异步化已落地：

```http
POST /names/tasks
GET /names/tasks/{task_id}
```

兼容策略：

- `/names/generate` 继续保留，用于兼容旧前端同步调用。
- 新前端建议改为 `POST /names/tasks` 提交任务，再轮询 `GET /names/tasks/{task_id}`。
- 任务状态为 `pending / running / success / failed`。
- 成功后任务会关联 `record_id`、`thread_id` 和 `result_data`。

部署侧已新增：

- `name_worker.py`
- `email_worker.py`
- `docker-compose.yml` 中新增 `rabbitmq`、`name_worker`、`email_worker` 服务。

数据库侧已新增：

- `name_tasks` 表。
- Alembic 迁移：`a4c9e2d7f6b1_add_name_tasks.py`。

后续重点：

- 执行 `alembic upgrade head` 以创建 `name_tasks` 表。
- 确认 RabbitMQ 在部署环境稳定启动。
- 为 `rag_document_queue`、`name_generation_queue`、`email_queue` 补充重试、死信队列和人工补偿策略。
- 前端逐步从同步起名切换到异步轮询。

## 1. 项目定位

本项目定位为一个面向 C 端与 B 端的 AI 智能命名平台，不只是简单的“起名字工具”，而是围绕“命名生成、可用性校验、品牌视觉、人工专家服务、开放 API、增长分销”构建的综合型平台。

第一阶段以 MVP 为目标，优先打通用户从注册登录到生成名字、多轮优化、保存历史、企业知识库增强和基础域名校验的完整闭环。

截至 2026-07-01，第一阶段 MVP 已完成并通过测试。后续进入第二阶段，重点从“核心闭环可用”升级为“校验矩阵增强、知识库管理完善、商业化基础预留和前端体验优化”。

长期目标是逐步扩展为：

- 面向个人用户的宝宝/个人起名平台
- 面向创业者和企业客户的品牌命名与核名平台
- 面向宠物、虚拟 IP、游戏、小说等场景的趣味命名工具
- 面向 B 端开发者的命名 API 服务平台
- 结合人工专家、社区投票和分销体系的平台型产品

核心目标：生成高质量、可解释、可持续优化的名字。

支持方向：

- 个人/宝宝起名
- 企业/品牌起名
- 宠物/虚拟 IP 起名
- 游戏角色/NPC 起名
- 小说角色/地名起名

第一阶段已完成：

- 三类起名全部可用
- 统一 AI 输出结构
- 支持多轮反馈
- 支持历史记录保存

- 八字五行可选输入

- 不同风格命名模板

### 模块二：数字资产全网校验矩阵 Validation Matrix

### 第一阶段免费次数规则

当前规则：

- 新用户注册后默认拥有 3 次免费生成机会
- “首次生成一个起名任务”扣 1 次
- 基于同一个 `thread_id` 的多轮反馈暂不扣次数
- 次数不足时，不允许新建起名任务
- 查看历史记录不扣次数
- 上传企业知识库不扣次数

这样可以鼓励用户深度体验一次完整流程，同时便于后续接入付费套餐。

## 6. 数据表规划

以下是面向长期规划的数据表建议。第一阶段已实现核心表，第二阶段建议围绕校验矩阵、候选名字增强、知识库管理和订单权益基础继续扩展。

### 6.1 用户表 users

用途：保存用户基础信息和登录身份。

建议字段：

- `id`：用户 ID
- `email`：邮箱，唯一
- `username`：用户名
- `password_hash`：密码哈希
- `free_quota`：剩余免费次数，默认 3
- `status`：用户状态，正常/禁用
- `created_at`：创建时间
- `updated_at`：更新时间

说明：当前项目已有用户表，可以在现有表基础上扩展免费次数字段，或者将次数放到独立账户权益表中。

### 6.2 使用记录表 usage_records

用途：记录用户每次消耗次数的行为，为后续会员、订单、统计做准备。

建议字段：

- `id`
- `user_id`
- `record_id`：关联起名记录 ID，可为空
- `usage_type`：使用类型，如 name_generate、report_generate、visual_generate
- `cost_count`：消耗次数
- `before_quota`：扣减前次数
- `after_quota`：扣减后次数
- `created_at`

说明：第一阶段用于记录免费 3 次的扣减过程。后续可以接会员权益和付费次数包。

### 6.3 起名记录表 name_records

用途：保存每次起名任务，是第一阶段最关键的业务表。

建议字段：

- `id`
- `user_id`
- `category`：人名/企业名/宠物名
- `title`：记录标题，可用第一个候选名或用户输入摘要
- `input_data`：用户输入参数 JSON
- `result_data`：AI 返回结果 JSON
- `thread_id`：LangGraph 会话 ID
- `status`：生成中/成功/失败
- `is_deleted`：是否删除
- `created_at`
- `updated_at`

说明：历史记录、高级报告、专家精批、收藏、订单都可以基于它扩展。

### 6.4 起名反馈表 name_feedbacks

用途：保存用户对某次起名记录的多轮修改意见。

建议字段：

- `id`
- `record_id`
- `user_id`
- `thread_id`
- `feedback_text`
- `result_data`：本轮反馈后的结果 JSON
- `created_at`

说明：如果第一阶段想简单一些，也可以只更新 `name_records.result_data`。但长期看，单独保存反馈历史更利于回溯和报告生成。

### 6.5 候选名字表 name_candidates

用途：结构化保存每个候选名字，便于收藏、投票、专家点评和视觉生成。

建议字段：

- `id`
- `record_id`
- `name`
- `reference`：出处/灵感来源
- `moral`：寓意
- `reason`：推荐理由
- `domain`：推荐域名
- `domain_status`：域名状态
- `score`：综合评分，可选
- `is_selected`：用户是否选中
- `is_favorite`：用户是否收藏
- `created_at`

说明：第一阶段可以先只保存 JSON。第二阶段若要做收藏、选中、对比和评分，建议拆出此表。

### 6.6 知识库文件表 knowledge_files

用途：记录企业知识库上传文件和处理状态。

建议字段：

- `id`
- `user_id`
- `filename`
- `file_path`
- `file_type`
- `file_size`
- `status`：pending/processing/completed/failed
- `error_message`

## 10. 关键产品规则

### 免费次数规则

- 新用户默认 3 次免费生成机会
- 首次生成新任务扣 1 次
- 多轮反馈暂不扣次数
- 历史查看不扣次数
- 知识库上传不扣次数
- 次数不足时禁止新建起名任务

### 起名记录规则

- 每次首次生成创建一条起名记录
- 起名记录保存输入参数、输出结果、类别、thread_id
- 多轮反馈可以更新最新结果，同时保存反馈历史
- 用户只能访问自己的记录

### 企业知识库规则

- 上传文件后立即返回上传成功和处理中状态
- 文件解析由 RabbitMQ 消费者异步完成
- 企业起名时只检索当前用户自己的知识库
- 第二阶段支持文件列表、删除、重建和失败重试

### 候选名字规则

- 每条起名记录可以包含多个候选名字
- 第二阶段支持收藏、选中、评分和对比
- 用户选中的名字可作为后续报告、视觉生成、专家精批和订单的业务入口

### 校验矩阵规则

- 第一阶段只做基础 `.com` 域名校验
- 第二阶段扩展为多后缀域名校验
- 商标和社媒风险先以结构化结果展示为主
- 风险结果只作为辅助判断，不应向用户承诺法律层面的绝对可用

## 11. 后续风险与注意事项

### 安全风险

第一阶段已完成后，需要优先处理配置安全问题。当前项目中存在一些配置直接写在代码中的情况，例如：

- 数据库账号密码
- 邮箱授权码
- JWT Secret
- 大模型 API Key

第二阶段建议迁移到环境变量或配置文件，并避免提交到公开仓库。

### 接口风险

后续商标、社媒、视觉生成等能力大概率依赖第三方付费接口，需要提前评估：

- 接口价格
- 调用频率限制
- 查询准确性
- 法律合规风险
- 用户隐私风险

第二阶段可以先做接口适配层、模拟数据和前端展示，不急于绑定具体付费供应商。

### 产品复杂度风险

项目长期规划较大，建议始终按阶段推进：

1. 第一阶段：核心起名闭环，已完成
2. 第二阶段：校验矩阵、知识库管理、候选名字增强、商业化基础
3. 第三阶段：付费、报告、视觉生成
4. 第四阶段：专家、社区
5. 第五阶段：开放平台、分销体系

不要在第二阶段同时铺开所有商业化和平台化模块。
