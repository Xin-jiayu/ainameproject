# AI 智能命名平台项目规划文档

## 1. 项目定位

本项目定位为一个面向 C 端与 B 端的 AI 智能命名平台，不只是简单的“起名字工具”，而是围绕“命名生成、可用性校验、品牌视觉、人工专家服务、开放 API、增长分销”构建的综合型平台。

第一阶段以 MVP 为目标，优先打通用户从注册登录到生成名字、多轮优化、保存历史、企业知识库增强和基础域名校验的完整闭环。

长期目标是逐步扩展为：

- 面向个人用户的宝宝/个人起名平台
- 面向创业者和企业客户的品牌命名与核名平台
- 面向宠物、虚拟 IP、游戏、小说等场景的趣味命名工具
- 面向 B 端开发者的命名 API 服务平台
- 结合人工专家、社区投票和分销体系的平台型产品

## 2. 当前已经完成的能力

### 2.1 后端基础能力

后端使用 FastAPI 开发，当前已经具备以下能力：

- 用户注册
- 邮箱验证码发送
- Redis 保存验证码
- 用户登录
- JWT 鉴权
- MySQL 用户数据存储
- FastAPI 路由分层
- SQLAlchemy 异步数据库连接
- Pydantic 请求与响应模型

### 2.2 AI 命名能力

当前已经支持三类起名：

- 人名/宝宝起名
- 企业/品牌起名
- 宠物/IP 起名

已经具备：

- LangGraph 工作流
- DeepSeek 大模型调用
- 结构化输出
- 多轮反馈优化
- 基于 `thread_id` 的上下文记忆
- 每次返回多个候选名字
- 名称、出处、寓意、域名、域名状态等字段

### 2.3 企业知识库能力

当前企业名模块已经接入 RAG 知识库能力：

- 用户上传企业资料文件
- 后端保存上传文件
- RabbitMQ 接收文件处理任务
- 消费者按任务逐个处理文件
- 文件内容进入用户专属向量知识库
- 企业起名时根据用户知识库进行检索增强

当前架构方向是正确的：上传接口只负责保存文件和投递任务，耗时解析由 RabbitMQ 消费者异步处理。

### 2.4 数字资产校验能力

当前已经具备基础域名校验：

- 企业名生成时生成 `.com` 域名建议
- 调用 whois 查询域名是否可注册
- 返回域名注册状态

后续将在此基础上扩展为数字资产全网校验矩阵。

### 2.5 前端基础能力

前端使用 uni-app / Vue3 开发，当前已经具备：

- 登录页面
- 注册页面
- 首页起名页面
- 人名、企业名、宠物名切换
- 企业知识库上传入口
- 首次生成名字
- 基于反馈再次生成
- token 本地保存
- 请求时自动携带 Bearer Token
- 结果卡片展示

## 3. 产品模块划分

结合长期规划，项目整体建议划分为以下模块。

### 模块一：多维度 AI 命名引擎 Core Engine

核心目标：生成高质量、可解释、可持续优化的名字。

支持方向：

- 个人/宝宝起名
- 企业/品牌起名
- 宠物/虚拟 IP 起名
- 游戏角色/NPC 起名
- 小说角色/地名起名

第一阶段重点：

- 三类起名全部可用
- 统一 AI 输出结构
- 支持多轮反馈
- 支持历史记录保存
- 提升 prompt 质量

长期扩展：

- 八字五行可选输入
- 音韵学分析
- 国学典故深度报告
- 品牌调性分析
- 行业竞品风格分析
- 不同风格命名模板

### 模块二：数字资产全网校验矩阵 Validation Matrix

核心目标：名字不仅好听，还要能用。

第一阶段重点：

- `.com` 域名可用性校验
- 结果中展示域名状态
- 为后续 `.cn`、`.ai`、商标、社媒校验预留结构

长期扩展：

- 自动生成全拼域名
- 自动生成首字母域名
- `.com` / `.cn` / `.ai` 多后缀检测
- 商标注册风险预估
- 指定商标类目查询
- 微信公众号重名风险
- 抖音、小红书等平台重名风险
- 给出综合可用性评分

### 模块三：多模态品牌视觉生成 AIGC Visuals

核心目标：从名字扩展到品牌冷启动服务。

可支持能力：

- 用户选定名字后生成 Logo 概念图
- 自动生成品牌 Slogan
- 自动生成名片排版
- 自动生成品牌色建议
- 自动生成视觉风格说明

第一阶段暂不实现，只在数据结构和产品路径上预留。

长期可对接：

- Midjourney
- Stable Diffusion
- OpenAI 图像接口
- 其他图像生成服务

### 模块四：人工专家与社区众包 Marketplace

核心目标：通过真人服务提高平台上限和客单价。

可支持能力：

- 国学专家精批
- 品牌咨询师精批
- 专家订单
- 专家工作台
- 专家报告
- 用户将候选名发布到社区投票
- 网友投票与评论建议

第一阶段暂不实现。

长期需要配套：

- 订单系统
- 专家入驻
- 审核机制
- 服务状态流转
- 结算体系
- 内容风控

### 模块五：B 端开放平台 API PaaS

核心目标：将命名能力包装为 API 出售。

可支持场景：

- 游戏公司 NPC/角色自动命名 API
- 小说网站角色/地名生成 API
- 企业 SaaS 命名插件
- 开发者 Dashboard
- API Key 管理
- 调用次数统计
- Token 或次数计费

第一阶段暂不实现。

长期需要配套：

- API Key 表
- 调用日志表
- 计费系统
- 限流系统
- 开发者控制台
- 文档中心

### 模块六：增长与分销裂变体系 Growth

核心目标：通过邀请和分销降低获客成本。

可支持能力：

- 邀请好友注册赠送次数
- 用户邀请码
- 分销合伙人二维码
- 孕婴店、工商代办人员推广

第一阶段暂不实现，但用户表和使用记录表应避免阻碍后续扩展。

## 4. 第一阶段 MVP 范围

第一阶段不追求六大模块全部做满，而是完成核心闭环。

### 第一阶段必须完成

- 用户注册与登录
- 新用户免费试用 3 次
- 三类起名：人名、企业名、宠物名
- 首次起名扣减 1 次
- 多轮反馈优化
- 起名历史记录
- 历史详情查看
- 可基于历史继续优化
- 企业知识库上传
- RabbitMQ 异步处理文件
- 企业名使用知识库增强
- 基础 `.com` 域名校验
- 前端展示剩余免费次数
- 前端展示历史记录

### 第一阶段暂不实现

- 支付
- 会员套餐
- 专家市场
- 社区投票
- Logo 生成
- 商标真实接口
- 社媒真实接口
- B 端开放平台
- 分销结算

### 第一阶段免费次数规则

建议规则：

- 新用户注册后默认拥有 3 次免费生成机会
- “首次生成一个起名任务”扣 1 次
- 基于同一个 `thread_id` 的多轮反馈暂不扣次数
- 次数不足时，不允许新建起名任务
- 查看历史记录不扣次数
- 上传企业知识库不扣次数

这样可以鼓励用户深度体验一次完整流程，同时便于后续接入付费套餐。

## 5. 数据表规划

以下是面向长期规划的数据表建议。第一阶段可以先实现核心表，后续逐步扩展。

### 5.1 用户表 users

用途：保存用户基础信息和登录身份。

建议字段：

- `id`：用户 ID
- `email`：邮箱，唯一
- `username`：用户名
- `password_hash`：密码哈希
- `free_quota`：剩余免费次数，默认 3
- `status`：用户状态，正常/禁用
- `created_at`：创建时间
- `updated_at`：更新时间

说明：当前项目已有用户表，可以在现有表基础上扩展免费次数字段，或者将次数放到独立账户权益表中。

### 5.2 使用记录表 usage_records

用途：记录用户每次消耗次数的行为，为后续会员、订单、统计做准备。

建议字段：

- `id`
- `user_id`
- `record_id`：关联起名记录 ID，可为空
- `usage_type`：使用类型，如 name_generate、report_generate、visual_generate
- `cost_count`：消耗次数
- `before_quota`：扣减前次数
- `after_quota`：扣减后次数
- `created_at`

说明：第一阶段用于记录免费 3 次的扣减过程。后续可以接会员权益和付费次数包。

### 5.3 起名记录表 name_records

用途：保存每次起名任务，是第一阶段最关键的业务表。

建议字段：

- `id`
- `user_id`
- `category`：人名/企业名/宠物名
- `title`：记录标题，可用第一个候选名或用户输入摘要
- `input_data`：用户输入参数 JSON
- `result_data`：AI 返回结果 JSON
- `thread_id`：LangGraph 会话 ID
- `status`：生成中/成功/失败
- `is_deleted`：是否删除
- `created_at`
- `updated_at`

说明：历史记录、高级报告、专家精批、收藏、订单都可以基于它扩展。

### 5.4 起名反馈表 name_feedbacks

用途：保存用户对某次起名记录的多轮修改意见。

建议字段：

- `id`
- `record_id`
- `user_id`
- `thread_id`
- `feedback_text`
- `result_data`：本轮反馈后的结果 JSON
- `created_at`

说明：如果第一阶段想简单一些，也可以只更新 `name_records.result_data`。但长期看，单独保存反馈历史更利于回溯和报告生成。

### 5.5 候选名字表 name_candidates

用途：结构化保存每个候选名字，便于收藏、投票、专家点评和视觉生成。

建议字段：

- `id`
- `record_id`
- `name`
- `reference`：出处/灵感来源
- `moral`：寓意
- `reason`：推荐理由
- `domain`：推荐域名
- `domain_status`：域名状态
- `score`：综合评分，可选
- `is_selected`：用户是否选中
- `created_at`

说明：第一阶段可以先只保存 JSON。若后续要做投票、收藏、Logo 生成，建议拆出此表。

### 5.6 知识库文件表 knowledge_files

用途：记录企业知识库上传文件和处理状态。

建议字段：

- `id`
- `user_id`
- `filename`
- `file_path`
- `file_type`
- `file_size`
- `status`：pending/processing/completed/failed
- `error_message`
- `created_at`
- `updated_at`

说明：当前已有 RabbitMQ 任务处理机制，后续可以让消费者更新这个表的状态。

### 5.7 域名校验表 domain_checks

用途：保存候选名字对应的域名校验结果。

建议字段：

## 9. 关键产品规则

### 免费次数规则

- 新用户默认 3 次免费生成机会
- 首次生成新任务扣 1 次
- 多轮反馈暂不扣次数
- 历史查看不扣次数
- 知识库上传不扣次数
- 次数不足时禁止新建起名任务

### 起名记录规则

- 每次首次生成创建一条起名记录
- 起名记录保存输入参数、输出结果、类别、thread_id
- 多轮反馈可以更新最新结果，同时保存反馈历史
- 用户只能访问自己的记录

### 企业知识库规则

- 上传文件后立即返回上传成功和处理中状态
- 文件解析由 RabbitMQ 消费者异步完成
- 企业起名时只检索当前用户自己的知识库
- 后续支持文件列表、删除、重建和失败重试

## 10. 后续风险与注意事项

### 安全风险

当前项目中存在一些配置直接写在代码中的情况，例如：

- 数据库账号密码

- 调用频率限制
- 查询准确性
- 法律合规风险
- 用户隐私风险

### 产品复杂度风险

项目长期规划较大，建议始终按阶段推进：

1. 先做核心起名闭环

4. 再做视觉生成
